实施 Roadmap

## 目标

- 将当前分析得到的功能特性沉淀为后续实施的优先级清单
- 先解决影响稳定性与一致性的基础问题，再推进性能、维护性与架构演进

## 优先级原则

- P0：影响正确性、运行一致性、停机安全性或运维风险，优先处理
- P1：影响性能、维护效率、产品完整性，建议在 P0 后集中推进
- P2：面向中长期扩展能力与架构演进，可分阶段推进

## P0 基础治理

### 1. 统一写操作保护

- 目标
  - 在 ui 模式或只读场景下，所有写接口行为一致
- 现状问题
  - 部分写接口已有保护，但创建任务、更新任务、批量对象操作、重排、配置修改等入口仍存在不一致风险
- 建议动作
  - 梳理全部 REST 写接口
  - 统一抽象写保护中间层
  - 为 UI 模式补齐接口级测试
- 交付结果
  - 所有写操作在受限模式下返回一致错误语义

### 2. 统一下载根目录与文件暴露路径

- 目标
  - 确保真实下载目录与 `/files` 暴露目录一致
- 现状问题
  - 下载根目录已支持配置化，但文件浏览路径仍可能跟随旧默认目录
- 建议动作
  - 收敛统一的文件系统根目录来源
  - 让 API 文件访问与下载器目录使用同一配置
  - 增加配置覆盖场景测试
- 交付结果
  - 页面打开文件、聚合预览与实际下载落盘目录一致

### 3. 完善优雅停机

- 目标
  - 停机时安全结束 HTTP 服务、worker、队列与状态写回
- 现状问题
  - 当前停机流程偏轻，容易留下未收敛的下载状态或未完成的资源释放
- 建议动作
  - 为 HTTP Server 增加显式关闭流程
  - 等待 worker 退出并处理队列清理
  - 收敛 downloading 状态并确保必要 flush
- 交付结果
  - 服务停止后状态一致、资源释放明确、重启恢复更稳定

## P1 性能与维护性

### 4. 优化聚合查询路径

- 目标
  - 降低对象数量增大时的聚合接口开销
- 现状问题
  - 目前聚合接口偏向全量拉取后再过滤、排序、分页
- 建议动作
  - 拆分聚合查询与视图组装职责
  - 优先下推筛选、排序、分页逻辑
  - 为大样本数据增加基准与回归测试
- 交付结果
  - 聚合接口在对象规模上升后仍保持可接受响应时间与内存占用

### 5. 补齐配置接口与界面配置面

- 目标
  - 让 API 与 UI 能完整表达后端已有配置能力
- 现状问题
  - 后端已有 filesystem、http、proxy、progress、ffmpeg 等细粒度配置，但界面与接口暴露不完整
- 建议动作
  - 对齐配置结构与返回模型
  - 补充配置编辑入口与校验
  - 增加导入前校验与错误提示
- 交付结果
  - 用户可在管理端完整查看并维护关键下载配置

### 6. 拆分前端单文件页面

- 目标
  - 降低页面耦合度，提升可维护性与测试性
- 现状问题
  - 当前 UI 主要集中在单个大型 `index.html`
- 建议动作
  - 按任务页、聚合页、配置页、公共状态区拆分逻辑
  - 抽取 API 调用、状态管理与通用组件
  - 为关键交互补充端到端验证
- 交付结果
  - 页面结构清晰，新增功能与排查问题的成本下降

### 7. 明确安全边界并做最小加固

- 目标
  - 明确项目适用部署场景，并补齐基础安全防护
- 现状问题
  - API 无鉴权、SSE 来源宽松、文件目录可直接暴露
- 建议动作
  - 明确默认仅内网使用或引入基础鉴权
  - 收敛 SSE 与文件接口访问边界
  - 在 README 与部署文档中标注安全前提
- 交付结果
  - 部署风险更可控，用户对使用边界有清晰预期

## P2 架构演进

### 8. 拆分 Manager 职责

- 目标
  - 将调度、聚合、对象控制、配置治理拆成更清晰的服务边界
- 建议动作
  - 先抽离聚合服务与对象操作服务
  - 再收敛调度服务与配置治理服务
  - 保持 `Manager` 仅承担生命周期与组装职责
- 交付结果
  - 核心编排更清晰，后续迭代的变更影响面更可控

### 9. 沉淀任务模板与站点适配层

- 目标
  - 降低新增站点或调整解析逻辑的重复成本
- 建议动作
  - 抽出缓存恢复、详情解析、ready 判定、失败处理模板
  - 将解析逻辑逐步收敛到 adapter 边界
  - 建立快照测试与解析契约
- 交付结果
  - 新任务开发效率更高，站点变更后的修复范围更可控

### 10. 收敛对象扩展字段协议

- 目标
  - 规范 `Extra` 与 `Metadata` 的高频字段语义
- 建议动作
  - 建立字段清单与命名约束
  - 为常用字段增加结构化访问层
  - 逐步减少隐式 map 协议扩散
- 交付结果
  - 前后端与多任务之间的数据契约更稳定

## 建议推进顺序

- 第一阶段
  - 写操作保护一致性
  - 下载路径一致性
  - 优雅停机
- 第二阶段
  - 聚合查询性能
  - 配置接口补全
  - 前端页面拆分
  - 基础安全加固
- 第三阶段
  - Manager 拆分
  - 任务模板与站点 adapter 化
  - 对象扩展字段协议治理

## 里程碑判定

- M1 基础稳定
  - 受限模式下无越权写操作
  - 文件访问路径与下载目录一致
  - 停机与重启后状态收敛正常
- M2 维护增强
  - 聚合接口性能改善
  - 配置面完整可用
  - 前端结构明显解耦
  - 安全边界文档化并落地最小保护
- M3 架构演进
  - 核心编排职责拆分完成
  - 新任务接入路径标准化
  - 高频扩展字段具备统一约束
